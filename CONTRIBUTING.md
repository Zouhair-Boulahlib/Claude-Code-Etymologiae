# Contributing to Claude Code Etymologiae

Thanks for considering a contribution! This guide is built by practitioners, for practitioners.

## How to Contribute

### Share a Pattern
If you've discovered a workflow or technique that consistently works well:

1. Fork this repository
2. Create a branch: `git checkout -b pattern/your-pattern-name`
3. Add your content in the appropriate section
4. Submit a PR with a clear description of the pattern and when it applies

### Report an Anti-Pattern
Found something that looks like good advice but leads to trouble?

1. Open an issue describing the anti-pattern
2. Include a concrete example of when it fails
3. Suggest what to do instead

### Fix or Improve Existing Content
- Typos, clarity improvements, better examples — all welcome
- If restructuring a section, open an issue first to discuss

## Writing Guidelines

- **Be concrete.** Show code, show commands, show real scenarios.
- **Be honest.** If something doesn't work well, say so.
- **Be brief.** Respect the reader's time. One clear example beats three mediocre ones.
- **No fluff.** Skip the motivational intros. Get to the point.
- **Test your advice.** If you're recommending a workflow, use it yourself first.

## Content Structure

Each guide should follow this structure:

```markdown
# Title

> One-line summary of what this covers

## The Problem
What challenge does this address?

## The Pattern
How to do it, with examples.

## When to Use This
Specific scenarios where this applies.

## When NOT to Use This
Scenarios where this would be counterproductive.

## Examples
Real-world, copy-pasteable examples.
```

## Code of Conduct

Be professional. Be helpful. Assume good intent. We're all here to get better at our craft.
