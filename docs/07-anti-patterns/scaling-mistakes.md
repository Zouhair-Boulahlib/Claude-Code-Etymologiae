# Scaling Mistakes -- Anti-Patterns That Get Worse With Growth

> Problems that are minor at 3 developers become catastrophic at 30.

Most AI-assisted development anti-patterns are tolerable in small teams. A quirky CLAUDE.md, a copied prompt, an unreviewed diff -- these are small fires you can stomp out when three people share a desk. But teams grow. Codebases grow. And these small fires become structure fires that no amount of AI can extinguish.

This guide covers seven anti-patterns that scale badly. Each one looks harmless at first. None of them stay harmless.

---

## 1. The Snowflake CLAUDE.md

Each developer has their own personal conventions in CLAUDE.md that contradict each other.

### Real Scenario

Three developers on a small team each have opinions. Developer A adds this to the project CLAUDE.md:

```markdown
## Code Style
- Use functional components with arrow functions
- Prefer named exports
- Use `type` for all TypeScript type definitions
```

Developer B commits a change the next week:

```markdown
## Code Style
- Use function declarations for components (hoisting matters)
- Use default exports for components, named exports for utilities
- Use `interface` for object shapes, `type` for unions only
```

Developer C merges their branch, which has yet another version:

```markdown
## Code Style
- All components must be class-based for consistency with our legacy code
- Default exports everywhere
- Avoid TypeScript generics -- keep it simple
```

At 3 developers, this is an annoying merge conflict. At 20 developers, the codebase is a patchwork. Half the components are arrow functions with named exports and `type`. The other half are function declarations with default exports and `interface`. A few files still use class components. Every AI-assisted coding session produces output that matches whichever CLAUDE.md version was active when that developer last pulled.

### Why It Gets Worse at Scale

Every new developer who joins copies the CLAUDE.md they see on their first day. If they happen to pull a branch with Developer B's version, that is their reality. AI generates code consistent with the instructions it sees -- so the inconsistency propagates automatically. At 30 developers across 5 teams, you can have 5 different coding styles all generated by the same AI tool, all technically "following the CLAUDE.md."

### The Fix

One CLAUDE.md, owned by the team lead or architect, reviewed like code.

```markdown
# .claude/CLAUDE.md (project root -- committed to main)

## Code Style (approved 2025-01-15, reviewed quarterly)
- Functional components using `function` declarations
- Named exports for all modules
- `interface` for object shapes, `type` for unions and intersections
- No class components in new code (legacy classes are OK to leave as-is)

## Ownership
- Changes to this file require approval from @platform-team
- PRs modifying CLAUDE.md must include rationale in the description
```

Developers who need personal preferences use `~/.claude/CLAUDE.md` for their home directory configuration -- things like editor keybindings, personal formatting preferences, or workflow habits. Project-level conventions live in the repo and go through code review.

## 2. The Abstraction Factory Factory (at Scale)

AI generates enterprise patterns for simple problems. One developer accepts it, others copy the pattern. Now the whole codebase is `AbstractServiceProviderFactoryRegistry`.

### Real Scenario

Developer A asks Claude Code to create a notification sender. AI generates:

```java
public interface NotificationChannel {
    void send(Notification notification);
}

public class EmailNotificationChannel implements NotificationChannel {
    @Override
    public void send(Notification notification) {
        // sends email
    }
}

public class NotificationChannelFactory {
    public static NotificationChannel create(String type) {
        return switch (type) {
            case "email" -> new EmailNotificationChannel();
            case "sms" -> new SmsNotificationChannel();
            default -> throw new IllegalArgumentException("Unknown channel: " + type);
        };
    }
}

public class NotificationService {
    private final NotificationChannelFactory factory;
    private final NotificationTemplateResolver templateResolver;
    private final NotificationPersistenceStrategy persistenceStrategy;

    // constructor injection, 40 more lines...
}
```

Developer A ships it. Works fine. The next week, Developer B needs to build a payment processor. They see the notification pattern and tell Claude Code: "Follow the same architecture as `NotificationService`." AI happily generates:

```java
public interface PaymentGateway { ... }
public class StripePaymentGateway implements PaymentGateway { ... }
public class PaymentGatewayFactory { ... }
public class PaymentService {
    private final PaymentGatewayFactory factory;
    private final PaymentValidationStrategy validationStrategy;
    private final PaymentPersistenceStrategy persistenceStrategy;
    // ...
}
```

By month three, every service in the codebase follows this pattern. The team has 47 factory classes, 32 strategy interfaces, and 15 "persistence strategies" that are all just wrappers around the same Hibernate calls. New developers spend their first week learning the abstraction taxonomy instead of the business domain.

What the codebase actually needed:

```java
public class NotificationService {
    private final EmailClient emailClient;

    public void sendEmail(String to, String subject, String body) {
        emailClient.send(to, subject, body);
    }
}
```

One class. One dependency. When you actually need SMS support, add a method or a second service. Do not build an extension framework for a problem you have today.

### Why It Gets Worse at Scale

Patterns are viral. Once a team establishes a precedent, AI replicates it. Developers reference it in prompts. Code reviewers approve new code because "it matches the existing pattern." Nobody questions whether the pattern was right in the first place. At 30 developers, the abstraction overhead dominates the codebase -- and refactoring out 200 unnecessary factories is a project nobody wants to take on.

### The Fix

Add an explicit complexity gate to your CLAUDE.md:

```markdown
## Architecture Rules
- No interface unless there are 2+ concrete implementations TODAY (not "someday")
- No factory unless object creation logic is genuinely complex
- No strategy pattern unless the algorithm actually varies at runtime
- When in doubt, write a plain function. Refactor to patterns when the need is proven.
```

And in code review: reject the first unnecessary factory. If you let one through, you have established the pattern for every service that follows.

## 3. Configuration Drift

AI fixes a bug on staging by changing a config value. Nobody updates prod config. Works on staging, crashes on prod at 2 AM.

### Real Scenario

A developer debugging a timeout on staging asks Claude Code to fix it. AI suggests bumping the connection pool size:

```yaml
# application-staging.yml (Spring Boot)
spring:
  datasource:
    hikari:
      maximum-pool-size: 20       # was 10, AI changed to fix timeouts
      connection-timeout: 60000   # was 30000
  redis:
    timeout: 10000                # was 5000
```

Fix works on staging. Developer commits. Nobody touches `application-prod.yml`:

```yaml
# application-prod.yml -- still has old values
spring:
  datasource:
    hikari:
      maximum-pool-size: 10
      connection-timeout: 30000
  redis:
    timeout: 5000
```

Two weeks later, prod traffic spikes. Same timeout bug. 2 AM page. The on-call engineer burns an hour rediscovering the fix that was already applied to staging.

This gets worse with `.env` files:

```bash
# .env.staging -- updated by AI last Tuesday
DATABASE_POOL_SIZE=20
REDIS_TIMEOUT=10000
FEATURE_FLAG_NEW_CHECKOUT=true
API_RATE_LIMIT=500

# .env.production -- nobody updated this
DATABASE_POOL_SIZE=10
REDIS_TIMEOUT=5000
# FEATURE_FLAG_NEW_CHECKOUT doesn't exist here at all
API_RATE_LIMIT=100
```

And with Docker Compose:

```yaml
# docker-compose.staging.yml -- AI added health check
services:
  api:
    environment:
      - NODE_ENV=staging
      - MAX_WORKERS=4
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

# docker-compose.production.yml -- no health check, different worker count
services:
  api:
    environment:
      - NODE_ENV=production
      - MAX_WORKERS=2
    # no healthcheck -- container restarts go undetected
```

### Why It Gets Worse at Scale

At 3 developers, someone remembers that they changed staging config last week. At 30 developers across multiple services, config changes happen daily. Nobody has a mental model of which environment has which values. AI happily fixes things in whatever environment you point it at, with no awareness that other environments exist.

### The Fix

Make config drift structurally impossible:

```yaml
# config/base.yml -- shared defaults
spring:
  datasource:
    hikari:
      maximum-pool-size: ${DB_POOL_SIZE:10}
      connection-timeout: ${DB_CONN_TIMEOUT:30000}
  redis:
    timeout: ${REDIS_TIMEOUT:5000}
```

Store environment-specific overrides in a single, auditable place -- a secrets manager, a deployment manifest, or at minimum a single `.env` file per environment that is version-controlled and reviewed:

```bash
# environments/staging.env
DB_POOL_SIZE=20
DB_CONN_TIMEOUT=60000
REDIS_TIMEOUT=10000

# environments/production.env
DB_POOL_SIZE=20          # <-- same as staging after review
DB_CONN_TIMEOUT=60000
REDIS_TIMEOUT=10000
```

Add to your CLAUDE.md:

```markdown
## Configuration
- NEVER change config for one environment without updating all environments
- All config changes require updating environments/staging.env AND environments/production.env
- If you change a timeout, pool size, or feature flag, mention all affected environments in the commit message
```

## 4. The Unreviewed AI Debt

Team ships AI-generated code without proper review because "it works." Months later, nobody understands the codebase.

### Real Scenario

A team of 8 adopts Claude Code. Velocity doubles overnight. Over three months:

```
Month 1:
  Lines of code generated by AI:   12,000
  Lines actually reviewed line-by-line: 9,000  (75%)
  Lines skimmed or rubber-stamped:      3,000  (25%)
  Bugs found in production:             2

Month 2:
  Lines of code generated by AI:   18,000
  Lines actually reviewed line-by-line: 10,000 (56%)
  Lines skimmed or rubber-stamped:      8,000  (44%)
  Bugs found in production:             5

Month 3:
  Lines of code generated by AI:   25,000
  Lines actually reviewed line-by-line: 8,000  (32%)
  Lines skimmed or rubber-stamped:      17,000 (68%)
  Bugs found in production:             14
```

The inflection point is around 40-50% unreviewed code. Below that threshold, the team can still reason about the codebase because most of it passed through a human brain. Above it, large sections of the code are black boxes -- they work, but nobody can explain why they are structured the way they are.

By month 4, a new developer joins and asks "why does `OrderService` make three separate database calls instead of a join?" Nobody knows. The AI generated it. The reviewer approved it because tests passed. The answer is buried in a prompt that no longer exists.

By month 6, a critical bug requires modifying the payment flow. The developer assigned to it opens `PaymentProcessor.ts` -- 400 lines of AI-generated code with no comments explaining the business logic. The original developer has moved to another team. The tests verify implementation details, not business rules. The developer is afraid to change anything because they cannot predict what will break.

### Why It Gets Worse at Scale

Review fatigue is real. As AI output volume increases, the percentage that gets reviewed decreases -- not because people are lazy, but because there are only so many hours in the day. The unreviewed code compounds. At 30 developers all generating code at 2x speed, the codebase grows faster than any human can comprehend.

### The Fix

Cap the velocity. This sounds counterintuitive, but it works:

```markdown
# Team Policy: AI Code Review Standards

1. Every AI-generated diff gets the same review as human-written code
2. If a PR is too large to review carefully, it is too large -- split it
3. AI-generated code must include comments explaining WHY, not just WHAT
4. Maximum PR size: 400 lines of changed code (including AI-generated)
5. Any function over 30 lines must have a comment block explaining its purpose
```

Track the ratio. If your team is generating more code than it can review, you are building debt, not features.

```markdown
## CLAUDE.md Addition
- When generating code, add a brief comment at the top of each function
  explaining the business purpose, not the implementation
- Example: "// Calculates shipping cost using zone-based rates from the
  //          2024 carrier contract. See SHIP-1234 for rate table."
```

## 5. The Golden Path Dictatorship

One senior developer mandates a specific AI workflow that does not fit every team's needs. Backend team forced to use frontend team's prompt patterns.

### Real Scenario

The VP of Engineering reads a blog post about AI-assisted development. They appoint a senior frontend developer as the "AI Champion." That developer creates a company-wide CLAUDE.md:

```markdown
## Mandatory AI Workflow (all teams)
- Always start with a component spec before writing code
- Use TDD: write tests first, then ask AI to implement
- All prompts must follow the format: "As a [role], I need [feature] so that [benefit]"
- Always ask AI to generate Storybook stories alongside components
- Run `npm test` after every AI-generated change
```

The backend team building Kafka consumers and database migrations is now required to write "component specs" and "Storybook stories" for their message handlers. The data team writing ETL pipelines in Python is told to use `npm test`. The infrastructure team writing Terraform modules has to frame their prompts as user stories.

The result: teams either ignore the mandate (creating shadow workflows that management does not know about) or follow it cargo-cult style (writing meaningless "component specs" for cron jobs to satisfy compliance).

### Why It Gets Worse at Scale

At 3 developers working on one stack, a shared workflow makes sense. At 30 developers across frontend, backend, data, and infrastructure teams, a single workflow is a straitjacket. The more rigid the mandate, the more teams work around it -- and workarounds are invisible to leadership until something breaks.

### The Fix

Create a layered system: shared principles at the top, team-specific practices at the bottom.

```markdown
# Company-Wide: .claude/CLAUDE.md (principles only)

## AI Usage Principles
- All AI-generated code must be reviewed with the same rigor as human code
- AI output must follow existing project conventions, not introduce new ones
- Security-sensitive code (auth, payments, PII) requires senior review
  regardless of whether AI generated it
- When AI suggests adding a dependency, verify it is actively maintained
  and has no known vulnerabilities
```

```markdown
# Frontend Team: frontend/.claude/CLAUDE.md

## Frontend AI Guidelines
- Generate components as functional components with hooks
- Always include unit tests using React Testing Library
- Prefer named exports
- When generating forms, include validation and error states
```

```markdown
# Backend Team: backend/.claude/CLAUDE.md

## Backend AI Guidelines
- Generate services as plain classes with constructor injection
- Always include integration tests for database operations
- Use parameterized queries -- never string interpolation for SQL
- When generating API endpoints, include input validation and auth checks
```

```markdown
# Data Team: data-pipelines/.claude/CLAUDE.md

## Data Pipeline AI Guidelines
- Generate pipeline steps as pure functions where possible
- Always include data validation at pipeline boundaries
- Log row counts at each transformation step
- When generating SQL, include EXPLAIN plan for queries touching >1M rows
```

Each team owns their workflow. Shared principles prevent chaos. Nobody is writing Storybook stories for Kafka consumers.

## 6. The Copy-Paste Prompt Epidemic

Someone shares a "magic prompt" in Slack. 15 people use it without understanding it. It generates subtly wrong code for different contexts.

### Real Scenario

A backend developer discovers a prompt that generates solid Express.js CRUD endpoints:

```
Generate a complete REST API endpoint for [resource].
Include: input validation with Joi, error handling middleware,
pagination with cursor-based navigation, rate limiting per user,
and OpenAPI documentation comments. Follow REST conventions strictly.
Use async/await with try/catch in every handler.
```

They post it in #engineering-tips: "This prompt is incredible, saves me 30 minutes per endpoint."

A Python developer on the team sees it and uses it verbatim for a FastAPI project:

```python
# AI generates Express-style code patterns in Python:

from fastapi import FastAPI
import joi  # <-- does not exist in Python

app = FastAPI()

@app.get("/users")
async def get_users():
    try:  # <-- unnecessary: FastAPI handles exceptions with exception handlers
        users = await db.find_all()
        return {"data": users, "cursor": encode_cursor(users[-1].id)}
    except Exception as e:
        return {"error": str(e)}, 500  # <-- wrong: FastAPI uses HTTPException,
                                        #     and tuples are not valid responses
```

The prompt assumes Express conventions: Joi for validation (does not exist in Python), manual try/catch in every handler (FastAPI has exception handlers), tuple responses with status codes (that is Flask, not FastAPI). The AI does its best to translate, but "its best" is a Frankenstein of Express patterns forced into FastAPI.

The correct FastAPI version looks nothing like this:

```python
from fastapi import FastAPI, HTTPException, Query
from pydantic import BaseModel

app = FastAPI()

class UserResponse(BaseModel):
    id: int
    name: str
    email: str

@app.get("/users", response_model=list[UserResponse])
async def get_users(cursor: str | None = Query(None), limit: int = Query(20, le=100)):
    users = await db.find_after_cursor(cursor, limit)
    if not users:
        raise HTTPException(status_code=404, detail="No users found")
    return users
```

Pydantic for validation (not Joi). Exception handlers (not try/catch). Type annotations for documentation (not JSDoc comments). The frameworks solve the same problems with completely different idioms.

### Why It Gets Worse at Scale

Prompts spread faster than understanding. One person posts a prompt. Fifteen people copy it. Five of them are using different frameworks, languages, or architectural patterns. Each gets output that is 80% correct and 20% subtly wrong -- and the 20% is different for each person. At scale, you end up with dozens of files containing framework-mismatched patterns, all traceable to a single Slack message.

### The Fix

Share context, not prompts. Instead of "use this prompt," document what the prompt is trying to achieve:

```markdown
# Team Wiki: API Endpoint Generation Guide

## What we want in every endpoint:
1. Input validation at the boundary
2. Proper error responses with meaningful messages
3. Pagination for list endpoints
4. Rate limiting for public endpoints
5. API documentation

## How to achieve this per framework:

### Express.js
- Validation: Joi or Zod middleware
- Errors: custom error-handling middleware
- Pagination: cursor-based, encode with base64
- Docs: JSDoc + swagger-jsdoc

### FastAPI
- Validation: Pydantic models + Query/Path params
- Errors: HTTPException + custom exception handlers
- Pagination: Query parameters with response_model
- Docs: automatic from type annotations

### Spring Boot
- Validation: Jakarta Bean Validation annotations
- Errors: @ControllerAdvice + @ExceptionHandler
- Pagination: Spring Data's Pageable
- Docs: springdoc-openapi annotations
```

Add to CLAUDE.md:

```markdown
## Prompting
- Do not copy prompts from Slack or external sources without adapting
  them to this project's framework and conventions
- If a prompt references a library or pattern not used in this project,
  rewrite it to use our equivalents
```

## 7. The "AI Will Fix It" Mentality

Team stops thinking about architecture because "we can always ask AI to refactor later." Technical debt compounds faster than AI can resolve it.

### Real Scenario

Sprint planning. The team needs to add multi-tenancy to a single-tenant application. Two approaches:

**Option A:** Spend two weeks designing the data model, adding tenant IDs to every table, updating every query, and modifying the auth middleware. Boring. Careful. Complete.

**Option B:** Just start building features. Hardcode tenant assumptions where needed. When we have 5 tenants, we will ask AI to refactor the whole thing. AI is great at refactoring. Ship now, fix later.

The team picks Option B. Here is what "fix later" looks like six months down the road:

```
Month 1: 3 hardcoded tenant assumptions. Easy to find, easy to fix.
Month 2: 12 assumptions. Scattered across services. Still manageable.
Month 3: 34 assumptions. Some in database queries, some in caching
         logic, some in background jobs. Getting harder.
Month 4: 67 assumptions. Several are load-bearing -- other code
         depends on the single-tenant behavior.
Month 5: 110+ assumptions. Some are implicit (code that works only
         because there is one tenant, without ever referencing tenancy).
         AI cannot find these because there is no keyword to search for.
Month 6: "AI, refactor this to multi-tenant."
         AI changes 40 files. Misses 15 implicit assumptions. Breaks
         the caching layer because cache keys assumed single tenant.
         Breaks 3 background jobs that query without tenant filters.
         Breaks the reporting dashboard that aggregates without
         tenant scoping.
         Estimated fix time: 4 weeks. Longer than Option A would have taken.
```

The cost curve is not linear. It is exponential:

```
Technical debt cost over time:

Month 1:  ████                          ~4 hours to fix
Month 2:  ████████                      ~2 days to fix
Month 3:  ████████████████              ~1 week to fix
Month 4:  ████████████████████████████  ~3 weeks to fix
Month 6:  ████████████████████████████████████████████████  ~2 months

The AI refactoring ceiling:
          ████████████████████████████████
          ^-- Beyond this point, AI cannot reliably refactor
              because too many assumptions are implicit
```

AI is good at mechanical refactoring -- renaming, extracting functions, moving files, updating imports. AI is poor at architectural refactoring -- changes that require understanding how the system behaves as a whole, what implicit assumptions exist, and how data flows across service boundaries.

### Why It Gets Worse at Scale

At 3 developers, technical debt grows slowly because there are only 3 people writing code. At 30 developers, debt grows 10x faster -- each developer adds assumptions, shortcuts, and "we'll fix it later" code. The refactoring surface area grows quadratically because the interactions between components grow quadratically. AI refactoring tools hit their ceiling much sooner because they cannot hold the entire system in context.

### The Fix

Architecture decisions are not refactorable. Treat them accordingly.

```markdown
## CLAUDE.md Addition

## Architecture Decisions
- Do not defer structural decisions with "we'll refactor later"
- Multi-tenancy, authentication model, data partitioning, and event
  architecture must be designed before implementation begins
- AI can help EXPLORE architectural options (ask it to compare approaches)
  but the DECISION is made by the team and documented in an ADR
- If you are adding a TODO comment that says "refactor this later,"
  stop and do it now or create a ticket with a deadline
```

Use AI for what it is good at -- exploring the design space:

```
"We need to add multi-tenancy to this single-tenant app. Compare these
approaches: row-level tenant IDs, schema-per-tenant, and database-per-tenant.
Our stack is PostgreSQL, Node.js, and Prisma. We expect 10-50 tenants with
very different data volumes. What are the tradeoffs?"
```

Then make the decision as a team, document it, and implement it properly. Do not ship the shortcut and hope AI will bail you out later.

---

## The Common Thread

All seven anti-patterns share a root cause: **treating AI-assisted development as a way to skip the hard parts of engineering.**

At small scale, you can skip the hard parts because one person holds the whole system in their head. At large scale, the hard parts -- consistency, architecture, review discipline, documentation -- are the only things preventing chaos.

AI makes individual developers faster. It does not make teams more coordinated. Coordination comes from shared standards, honest review, and architectural decisions made deliberately rather than deferred indefinitely.

Scale the discipline alongside the tooling, or the tooling will scale the problems.
